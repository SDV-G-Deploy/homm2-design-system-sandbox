How to use this manual

1 · Choose a material role Parchment reads, stone/night structures, gold commands, mana marks consequence.
2 · Assemble from components Use buttons, rails, parchment panels, plaques and ritual panels before local rules.
3 · Lock the recipe Recipe contracts decide hierarchy, optional layers and mobile collapse order before scene styling begins.
4 · Lock responsive behavior Use the recipe's first-screen budget, above-fold obligations and downgrade order before tuning page proportions.
5 · Compose locally Scene CSS sets layout, backdrop and dramaturgy. It must not invent button, rail or panel grammar.

System Layers

Tokens Color, type, spacing, shape, shadow and semantic roles. Tokens never decide screen choreography.
Shared components Buttons, rails, parchment surfaces, ritual frames, menu shell primitives and bounded asset slots.
Screen recipes The missing middle layer. Recipes define which components may appear together, in what hierarchy, and what collapses first on mobile.
Responsive contracts Per-recipe first-screen budgets, above-fold obligations, downgrade order, tap target floors and label legibility floors.
Page-local composition Backdrop, section order, copy rhythm, exact proportions and story-specific sequencing only.

Recipe Contract

One screen, one primary recipe Do not hybridize menu, dossier and ritual cores inside the same first-screen frame. Support patterns may appear, but one recipe owns hierarchy.
Required beats stay visible If a recipe requires a primary command or reading field, mobile collapse may not hide or miniaturize it before optional layers are reduced.
Bounded image policy still applies Art may only enter through approved slots like .menu-art-layer and must remain atmospheric, never structural or text-bearing.
Page CSS cannot rewrite system order Local composition may place and size recipe parts, but it must not promote tertiary rails above the primary command or reading task.

Menu Shell Contract

These classes are the reusable menu-screen anatomy. They define hierarchy and component ownership; page files may arrange them, but should not redefine their grammar.

Class Role Expected neighbors Hierarchy Must stay page-local
.menu-screen-shell Outermost framed menu stage and fallback surface. Wraps .menu-frame-shell; may contain .menu-art-layer. Backdrop foundation. Scene background, stage proportions, page-specific grid placement.
.menu-frame-shell Layout carrier for center, rails and subordinate state. Inside .menu-screen-shell; contains center mount and optional rails. Composition scaffold. Column counts, named grid areas, viewport-specific choreography.
.menu-center-mount Primary command and title anchor. Contains the central header, .command-slab and save slot. Primary focus. Scene title copy, crest choices, vertical offsets.
.command-slab Shared stack surface for major menu commands. Lives inside center mount; contains buttons or command links. Highest interactive weight. Command count, labels, separators and narrative ordering.
.menu-side-rail Secondary navigation, status or utility rail. Sibling of center mount inside frame shell. Subordinate support. Rail content model, local icons, collapsed ordering.
.menu-save-slot Compact state holder near the command stack. Usually follows command slab inside center mount. Persistent state, below commands. Save schema, labels, timestamp format.
.menu-route-index Low-volume route or chapter choices. Near center mount or below save slot; may use route badges. Tertiary navigation. Route taxonomy, active destination, disabled reasons.
.menu-art-layer Optional non-interactive image slot holder. Direct child of screen shell, behind frame content. Atmospheric support only. Whether a scene includes the layer, and which approved slots appear.
Shared grammar Surface roles, button weight, rail subordination, fallback art slots and focus behavior belong in shared CSS.
Allowed local override Pages may set grid areas, copy measure, exact ordering, backdrop treatment and scene-specific data density.
Not local Do not restyle command depth, invent new rail grammar, widen art slots or bake state text into images.

Compact Do / Do Not

Do - command hierarchyOne ceremonial command leads; quiet commands support or exit.
Do not - command hierarchyDo not place several gold commands with equal visual pressure in one slab.
Do - side rail densityKeep rails scan-friendly with compact groups and muted secondary state.
Do not - side rail densityDo not let a rail become a second command slab or a dense dossier.
Do - route indexUse route badges as small anchors beside HTML route labels.
Do not - route indexDo not make route choices compete with the main journey command.
Do - parchment fieldsUse parchment for reading, record keeping and comparison.
Do not - parchment fieldsDo not use parchment panels as decorative backing for every control.

Art Asset Layer

Generated imagery is allowed only as a bounded component slot. CSS still owns structure, states, copy, responsive behavior and fallback rendering.

CSS-only first Layout, rails, buttons, labels, states and adaptive compression never move into images.
Image-assisted slots Use generated art only for named slots like frame corners, edge tiles, parchment grain or route badges.
Fallback is mandatory Every slot must degrade to the existing component without losing hierarchy or readability.
Slot contract menu-frame-corner, menu-frame-edge-x, menu-frame-edge-y, route-badge-sprite. See the Art Slot + Fallback Rules for ownership and responsive checks.
Generation rule No readable text, no layout baked into art, no one-off ornaments without an owning component.
Pilot rule Frame corners and edge tiles are live. Route badges are now demonstrated as a small optional sprite slot.
Route badge sprite · optional asset layer

Pixel / Old-Style Motifs

Stepped edge cues Use subtle stepped cuts, frame joints and bevels — not retro cosplay.
Chunky CTA silhouette Buttons feel pressed or carved, not inflated pills.
Editorial-fantasy mix Borrow archival pacing, then ground it in fantasy materials and tokens.

Ornament Discipline

Use sigils as anchors Sigils, plaques and seals mark entry, ownership or consequence — not every box.
Frame before flourish Build hierarchy with rails, insets, headers and command depth before flourish.
Magic is a state, not wallpaper Glow and mana belong to focus, action and transformation only.

Page Recipes

MenuScreenRecipe

Menu-first entry screen for chapter selection, world entry and primary command routing.

Required components

  • .menu-screen-shell
  • .menu-frame-shell
  • .menu-center-mount
  • .command-slab

Optional layers

  • .menu-side-rail for status or utility
  • .menu-save-slot for persistent state
  • .menu-route-index for tertiary route choices
  • .menu-art-layer for bounded atmosphere

Hierarchy order

  1. Title and command slab own the first read.
  2. Save slot or immediate state follows the slab.
  3. Rails support, but never compete with, the center mount.
  4. Route index stays tertiary.
  5. Art remains background only.

Mobile collapse order

  1. Demote route index into a shorter list or below the fold.
  2. Compact rails into summaries before shrinking command labels.
  3. Keep the primary slab visible inside the first viewport.
  4. Condense the save slot, but keep it as one strip.

Responsive contract

  • First-screen budget: one 900px desktop viewport; one 844px mobile viewport with a next-section cue.
  • Above fold: title/context, primary command slab, one secondary command and immediate save/status when present.
  • Never use micro-text for command, utility, save-state or route-state labels.
  • Tap floor: 44px preferred, 40px compact minimum, 12px label minimum.

Forbidden combinations

  • Do not place a full dossier reading field inside the same first-screen shell.
  • Do not run multiple equal-weight command slabs side by side.
  • Do not let rails outrank the center mount through size, glow or density.

DossierReadingRecipe

Reading-first archive or dossier screen for records, comparison and retained lore.

Required components

  • .parchment-object as the main reading field
  • .archive-rail or equivalent reading spine

Optional layers

  • .note-card for marginal lore
  • .dossier-ledger-strip for compact metadata
  • Quiet utility actions outside the main parchment flow

Hierarchy order

  1. Parchment reading surface is the primary field.
  2. Archive rail or chapter spine guides orientation.
  3. Metadata strips summarize, but do not dominate.
  4. Notes remain secondary and skimmable.

Mobile collapse order

  1. Stack notes below the reading field.
  2. Compress metadata strips before reducing reading measure.
  3. Preserve one clear reading column before ornamental side matter.

Responsive contract

  • Above fold: dossier heading, first readable record/prose block and one spine cue.
  • Notes, decorative side matter and metadata compress before prose measure.
  • Never use micro-text for body prose, record labels or archive rail items.
  • Tap floor: utility and rail actions keep 40px compact minimum and 12px labels.

Forbidden combinations

  • Do not replace the reading surface with card mosaics.
  • Do not insert a dominant ceremonial CTA above the dossier opening.
  • Do not flank the parchment with equal-strength side rails on both sides.

RitualThresholdRecipe

Consequence-first threshold for commitment, initiation, warning or final choice.

Required components

  • .ritual-panel
  • One .btn-ceremonial
  • One quiet return or cancel action

Optional layers

  • .ritual-protocol or short consequence ledger
  • Wax seal or restrained mana cue

Hierarchy order

  1. Threshold title and consequence copy frame the decision.
  2. The ceremonial action leads.
  3. The quiet escape remains obvious but subordinate.
  4. Mana or seal cues support the threshold, never replace it.

Mobile collapse order

  1. Reduce decorative aura and side protocol first.
  2. Stack protocol below the ritual body.
  3. Keep the ceremonial action and escape readable at full tap size.

Responsive contract

  • Above fold: consequence statement, one ceremonial action and one quiet escape.
  • Aura, protocol and ledger detail compress before actions.
  • Never use micro-text for CTA, escape, warning or irreversible-choice copy.
  • Tap floor: both actions keep 44px preferred targets and 12px labels.

Forbidden combinations

  • Do not present more than one ceremonial action.
  • Do not combine route-board density with a ritual threshold shell.
  • Do not use mana glow as general section decoration.

RouteBoardRecipe

Stateful route progression board that shows what is current, locked, visited and next.

Required components

  • .menu-route-index or equivalent board list
  • .route-badge state marker for each path item

Optional layers

  • Compact lore annotation
  • Short progress/status strip
  • Quiet filter or sort control

Hierarchy order

  1. Current and next route state must read first.
  2. Badge state precedes lore.
  3. Route label and outcome hint follow the state.
  4. Annotations and filters stay secondary.

Mobile collapse order

  1. Condense annotations before shortening state labels.
  2. Move filters below the primary route list.
  3. Keep badge state and current path visibility intact.

Responsive contract

  • Above fold: current path, next path, badge states and route outcome hint.
  • Lore, filters, progress strips and secondary rows compress before current/next rows.
  • Never use micro-text for current/next, locked/available or badge-adjacent labels.
  • Tap floor: route rows keep 40px compact minimum and 12px labels.

Forbidden combinations

  • Do not turn every route row into a framed ritual panel.
  • Do not let lore blocks outweigh state labels.
  • Do not introduce multiple competing primary actions inside the board.

Accessibility Baseline

KeyboardAll links/buttons use :focus-visible. Do not remove it locally.
Mobile tap targetsCommands prefer 44px min-height; secondary controls may compact to 40px; labels stay 12px or larger.
ContrastQA gold-on-parchment and mana-on-dark after every theme override.
MotionReduced-motion is global; page CSS must not reintroduce mandatory animation.
