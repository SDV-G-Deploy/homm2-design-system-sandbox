## Findings

### 1. [medium] Pass 3 and Pass 4 drift away from Pass 2's stated fresh-contract priority, which weakens the system's own first-screen protection story

Evidence:

- Pass 2 says the highest-priority fresh contract passes are `SiteShellNavigationRecipe` first and `SideRailMetadataRecipe` second, before `ArticleHeroRecipe`, `DossierReadingRecipe`, and `FeedListingRecipe` because they protect first-view competition and support-panel creep: [recipe-family-map-2026-05-15.md](/root/.openclaw/workspace/homm2-design-system-sandbox/docs/recipe-family-map-2026-05-15.md:612)
- Pass 2 handoff to Pass 3 repeats that order explicitly: [recipe-family-map-2026-05-15.md](/root/.openclaw/workspace/homm2-design-system-sandbox/docs/recipe-family-map-2026-05-15.md:676)
- Pass 3 instead makes `ArticleHeroRecipe`, `DossierReadingRecipe`, `FeedListingRecipe`, `TeaserCardBlockRecipe`, then `SideRailMetadataRecipe` the critical-first focus, and pushes `SiteShellNavigationRecipe` to later follow-up: [content-contracts-and-budgets-2026-05-15.md](/root/.openclaw/workspace/homm2-design-system-sandbox/docs/content-contracts-and-budgets-2026-05-15.md:85)
- Pass 4 keeps that reordered priority and again places `SideRailMetadataRecipe` last in the tranche: [fixture-strategy-2026-05-15.md](/root/.openclaw/workspace/homm2-design-system-sandbox/docs/fixture-strategy-2026-05-15.md:155)

Why this matters:

Pass 2's logic is sound: shell navigation and side rails are cross-cutting failure sources. If they remain under-contracted while hero, dossier and feed fixtures advance, later generated pages can still fail in the first viewport because the surrounding shell and support context remain ambiguous. That creates a mismatch between the system's stated risk model and its actual hardening order.

Recommendation:

- either restore Pass 2's priority logic by explicitly scheduling `SiteShellNavigationRecipe` and `SideRailMetadataRecipe` before or alongside the first content-fixture tranche
- or amend Pass 2 and the constitution-facing handoff language to explain why the project intentionally pivoted toward content-family-first hardening

### 2. [medium] Pass 4 says SideRailMetadataRecipe should be proven against host recipes, but the canonical backlog still frames it mainly as an isolated fixture

Evidence:

- Pass 4 says `SideRailMetadataRecipe` is critical, but as a support family it should be proven against host recipes, not only in isolation: [fixture-strategy-2026-05-15.md](/root/.openclaw/workspace/homm2-design-system-sandbox/docs/fixture-strategy-2026-05-15.md:167)
- The canonical fixture list still defines a standalone `fixtures/side-rail-metadata-recipe.html` as the proof surface: [fixture-strategy-2026-05-15.md](/root/.openclaw/workspace/homm2-design-system-sandbox/docs/fixture-strategy-2026-05-15.md:331)
- The document itself leaves the real proof shape unresolved: should `SideRailMetadataRecipe` be proven mainly in isolation or as attached support inside hero, dossier and feed hosts?: [fixture-strategy-2026-05-15.md](/root/.openclaw/workspace/homm2-design-system-sandbox/docs/fixture-strategy-2026-05-15.md:454)

Why this matters:

For a support family, isolated proof can validate row budgets and summary behavior, but it cannot fully prove the core failure mode: whether the rail stays secondary to a host hero, reading field or listing surface. That means the current fixture plan risks proving the easier part of the pattern while leaving the hierarchy conflict under-tested.

Recommendation:

- keep the isolated side-rail fixture only as a bounded unit test surface
- add one explicit host-bound proof requirement, for example a dossier-hosted side-rail state and a feed-hosted side-rail state, before the family is considered regression-ready
- update the Pass 4 wording so the isolated fixture is named as partial proof, not the canonical end state
